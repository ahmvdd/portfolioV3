Architecture & Domain Design

Séparer clairement les responsabilités sans complexifier inutilement

L’objectif principal était de sortir d’un historique “monolithe spaghetti” : logique métier mélangée au rendu front et multiples sources de vérité côté contenu. J’ai posé une architecture en couches avec un domaine “Content” bien défini, des services d’accès uniformes et des points d’intégration explicites avec le reste du SI.

Domain & Services

Modèle de contenu unifié

Définition d’agrégats simples (Page, Bloc, Média) avec des règles métier explicites pour la publication, la validation et la historisation.

  • Contrats d’API clairs entre front, back et outils internes
  • Validation système des changements (au lieu de scripts ad‑hoc)
  • Stratégie de versionning des contenus pour rollback simple
Scope projet
Core CMS + API
Focalisation sur un noyau solide (contenu, publication, permissions) plutôt que sur la “feature factory”.

Ce que j’ai livré

  • Schémas de données & diagrammes d’architecture
  • Implémentation des premiers services (content, media, auth)
  • Documentation technique partageable avec les équipes
Migration

Migration progressive depuis l’ancien système : coexistence des deux sources de contenu, feature flags et bascule contrôlée par type de page.

  • Stratégie “page par page” pour limiter le risque
  • Scripts de migration répétables (idempotents)
  • Suivi des erreurs de migration via logs structurés

Contrats stables

Les fronts consomment un contrat stable, ce qui permet de faire évoluer la couche de persistance sans casser l’expérience utilisateur.

Reliability & Observability

Rendre visibles les défaillances plutôt que les cacher

L’un des points clés du projet était de rendre l’infrastructure prévisible : mieux vaut des erreurs explicites et tracées qu’un système silencieux qui “fonctionne la plupart du temps”.

Health

Mise en place de probes de santé, timeouts raisonnables et gestion systématique des timeouts côté clients.

Monitoring

Logs et métriques exploitables

J’ai structuré les logs autour de corrélations simples : id de requête, id de contenu, type d’action. Les erreurs sont regroupées par scénario métier, pas seulement par stacktrace.

log.info({
  event: "content_published",
  contentId, userId, channel, durationMs,
  result: "success"
});
  • Dashboard simple pour suivre les temps de publication
  • Alertes sur les erreurs récurrentes côté API
  • Traces utiles pour les retours produit/marketing
Permissions
type Role = "viewer" | "editor" | "publisher" | "admin";

function canPublish(user, content) {
  return user.role === "publisher" || user.role === "admin";
}

Modèle de rôles lisible

J’ai préféré un modèle RBAC simple, documenté pour les équipes métiers, plutôt qu’un système hyper granulaire impossible à maintenir.

Apprentissage

Ce projet m’a surtout appris à trouver le bon niveau d’abstraction : suffisamment structuré pour être solide, mais assez simple pour que les équipes puissent se l’approprier sans lire 200 pages de docs.

Delivery & Learnings

Ce que j’ai réellement livré sur ce projet

Au‑delà des schémas, l’enjeu était de livrer une base saine que les équipes pouvaient utiliser rapidement : un noyau CMS exploitable, des outils pour le faire évoluer et des garde‑fous pour éviter de retomber dans la dette technique précédente.

Livrables

Base produit en production

Premier incrément mis en service pour une partie du site, avec de vrais utilisateurs métiers et un support minimal des équipes techniques.

  • Back‑office simple pour créer / publier des pages
  • API de lecture consommée par le front existant
  • Documentation succincte pour l’onboarding des nouveaux devs
Décisions clés

Trade‑offs assumés

  • Privilégier un RBAC simple mais lisible plutôt qu’un système ultra granulaire
  • Accepter quelques duplications temporaires pendant la phase de migration
  • Limiter le nombre de technos pour faciliter la prise en main (un seul langage côté back)
Pour la suite

Évolutions prévues

Une fois le noyau stabilisé, les prochains incréments se concentrent sur :

  • Internationalisation des contenus (multi‑locale)
  • Workflows de validation plus riches (review, approbation)
  • Connecteurs dédiés vers d’autres systèmes internes (CRM, analytics)
Ce que ça dit de moi

Mon rôle dans ce type de sujet

J’aime prendre des problèmes flous (contenu éparpillé, dette historique) et les transformer en architectures concrètes, documentées, qui tiennent la route dans la durée tout en restant compréhensibles par les équipes produit et métier.
